## 一、设计思路与目标



### 1. 你的个性化定位：把“课程内容”变成“可验证的内核机制”

rcore/ucore 现有实验强项通常在：启动、内存基本管理、进程/线程、系统调用、简单文件系统等。但你提到的课程重点缺口——**调度算法、虚存管理算法、同步互斥、实时调度与优先级反转**——更像“理论模块”，如果不做成实验很容易停留在概念。



个性化教程可以明确一个核心目标：



> **每个理论点都必须对应一个：可实现的内核接口 + 可观测的指标 + 可复现实验脚本**
>
> （也就是“概念/机制—现象/实验—数据/对照”闭环）



### 2. 适合自己的学习方式（可写成报告里的“学习方式描述”）

建议你把学习方式写得具体，像“工程化学习者 / 数据驱动学习者”：



- **工程化**：先把接口做出来（trace、统计、hook），再迭代算法，不在一开始纠结最优设计。
- **数据驱动**：所有调度/内存/同步实验都必须输出可比较的指标（吞吐、平均等待、P99 延迟、缺页率、抖动次数、优先级反转持续时间等）。
- **对照学习**：每个实验都做 “baseline（原实验） vs 你的扩展” 的 A/B 对比。
  -

### 3. 初步设想与规划：把教程拆成“主线 + 选修支线”

你可以把个性化教程做成“主线必做 + 支线选做”，方便与你的兴趣对齐（比如你对实时系统/内存更感兴趣）。



- **主线（建议 1\~3 个小实验）**：围绕“可运行、可观测、可对比”的内核机制扩展
- **支线（建议 1\~3 个小实验）**：更深入的算法、优化与工程问题（如优先级继承/天花板协议、工作集等）

### 实验 ：同步互斥机制（从“能跑”到“公平/可证明不饿死”）



**目标**：把“锁”从概念变成可测系统：自旋锁、睡眠锁、信号量、条件变量、读写锁。



**实现点建议**：

- 从最简单的 `spinlock` 开始（关中断/原子指令）
- 再实现 `mutex`：拿不到锁就 sleep，释放时 wakeup
- `semaphore` 与 `condvar` 用来实现经典问题：生产者-消费者、读者写者、哲学家进餐
  -

**观测指标**：

- 锁竞争次数、平均持锁时间
- 上下文切换次数（睡眠锁 vs 自旋锁差异）
- 公平性：最大等待时间、是否出现 starvation
  -

**验收**：

- 每个同步原语都给一个“能失败的对照测试”（例如故意去掉 wakeup 或 unlock 顺序，测试就挂）
- 你写的正确版本必须通过压力测试

### 实验 ：虚拟内存管理算法（从“页表映射”到“缺页-置换-抖动”）



**目标**：补齐课程虚存算法：FIFO / LRU（近似）/ Clock / Working Set（简化）等。



**实现要点（典型路线）**：

1. 先做缺页异常处理（若已有则强化）
2. 加一个“页访问位/脏位”的软件维护（RISC-V 硬件位能用就用，不能就用软件标记）
3. 做页面置换：先 FIFO，再 Clock，再 LRU-approx
4. 做工作集/抖动检测：当缺页率飙升时打印告警或动态调参
   1.

**观测指标**：

- 缺页次数、缺页率（每秒/每千指令）
- 置换次数、回写次数（脏页）
- 抖动事件次数（thrashing）
  -

**验收**：

- userland 提供内存访问 pattern：顺序扫描、随机访问、局部性强/弱
- 输出算法对比数据

### 实验 ：实时调度 + 优先级反转（课程里最容易“只会背定义”的部分）



**目标**：把 RMS/EDF + 优先级反转 + 继承机制做成可复现实验。



**实现路线**：

- 先支持“任务优先级 + 周期/截止期”的任务模型（哪怕只在内核里模拟也行）
- 实现 RMS 或 EDF 的一个（建议 EDF 更直观可观察）
- 构造经典优先级反转场景：H（高优先级）等待 L 持有的锁，M（中优先级）抢占 L，导致 H 卡住
- 实现**优先级继承 PI**：L 临时提升到 H 的优先级直到释放锁
  -

**观测指标**：

- 反转持续时间（H 等待时长）
- deadline miss 次数（实时任务是否错过截止期）
- 有/无 PI 的对照结果（这是你报告里很亮眼的定量对比）
  -

**验收**：

- 固定脚本重复运行 N 次，输出平均值与方差（证明可复现）

## 三、与 AI 合作的实现过程（你报告里最重要的“方法论”部分）



AI是你的合作伙伴（亦师亦友），可把自己定位成“产品经理 + 架构师 + 研发工程师”，AI 是“代码生成器 + 参谋 + 测试顾问”。可以把 AI 的作用分成 5 类，并写清楚“怎么问、怎么验、怎么纠错”，这样很像真正的软件工程研发流程。最终的效果是希望你对操作系统的机制理解更深刻，能写出更健壮的代码。



### 1) 用 AI 做“课程内容 → 可实现接口”的转换

**典型交互方式**：

- 让 AI 把某算法拆成“内核需要哪些 hook 点”
- 让 AI 生成伪代码 + 数据结构建议 + 时间复杂度
  -

**示例提示词（你可直接放报告里）**：

- “请把 MLFQ 的实现拆成内核事件回调：on\_tick/on\_block/on\_wakeup，并给出队列设计与降级/升级规则。”
- “请给出优先级继承实现时，在 mutex lock/unlock 需要维护哪些字段、如何避免死锁与优先级传播链条问题。”

### 2) 用 AI 做“代码审阅 + 潜在竞态检查”

**关键点**：报告里要写你如何“验证 AI 的建议不是瞎的”：

### 3) 用 AI 辅助调试：从症状到假设再到定位

2. 收集证据（trace、断言、统计）

## 四、学习效果评估（定量 + 定性 + 与原教程对比）



这一部分建议你按“三层评估”写，既有数据也有主观反思。



### 1) 定量指标（建议你选 6\~10 个）

**覆盖度**（课程内容覆盖）：

- 原实验覆盖：启动/内存基础/进程/系统调用/FS（举例）
- 你的扩展覆盖：+ 调度算法（≥3种）、+ 页面置换（≥2种）、+ 同步原语（≥3种）、+ 实时调度与 PI（1套）
  -

**可观测性**：

- 原实验：日志分散、缺少统一指标
- 你的教程：统一 trace + 每实验必须输出指标（CSV/JSON）
  -

**可复现性**：

- 原实验：功能验收为主
- 你的教程：脚本化 workload + 重复运行统计方差
  -

**性能/行为对比（示例）**：

- MLFQ vs RR：交互延迟 P99 降低多少
- Clock vs FIFO：缺页率降低多少
- PI 开启前后：优先级反转持续时间下降多少、deadline miss 减少多少
  -

> 你不需要“结果很漂亮”，只要你有“方法可复现”，就很加分。



### 2) 定性评估（能力提升画像）

可以写成三点：

- **系统理解提升**：从“知道概念”到“知道它在内核里长什么样、哪里会出 bug”
- **工程能力提升**：trace/测试/回归/性能指标体系
- **抽象能力提升**：把不同算法统一到一个接口框架（可插拔 scheduler、统一 VM 统计）
  -

也可以诚实写下降/代价：

- 写实验花时间，导致刷题/看论文时间减少
- 为了可观测性引入日志开销，需要开关与分级
  -

### 3) 与本校现有实验教程的对比分析（可写成表格思路但用文字也行）

对比维度建议：

- 覆盖范围（课程点覆盖）
- 教学结构（是否机制-现象-数据闭环）
- 测试体系（是否脚本化、是否有压力测试）
- 可扩展性（是否可插拔、是否便于新增算法）
- 反馈速度（debug 是否更快、是否有可视化统计）
