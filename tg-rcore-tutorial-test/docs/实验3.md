## **一、问题分析：从文本程序到图形游戏的跃迁**



### **1.1 tg-ch3 的能力边界**



tg-ch3 已经实现了：

- ✅ 多道程序并发执行（多个任务同时驻留内存）
- ✅ 抢占式调度（时钟中断实现时间片轮转）
- ✅ 协作式调度（yield 系统调用主动让出 CPU）
- ✅ 标准系统调用（write、exit、yield、clock\_gettime）

**但存在限制**：

- ❌ 只能通过 UART 控制台输出文本
- ❌ 没有图形显示能力
- ❌ 没有键盘输入支持（只有非阻塞 UART 读取）
- ❌ 用户程序无法直接访问硬件（framebuffer、GPU）

### **1.2 用户态贪吃蛇游戏的需求**



贪吃蛇游戏需要：

\- 🎨 **图形渲染**：在屏幕上绘制蛇、食物、网格、分数

\- ⌨️ **键盘输入**：实时响应方向键（i/j/k/l）控制蛇的移动

\- 🔄 **高帧率更新**：约 5 FPS（每 200ms 更新一次）

\- 📊 **状态管理**：蛇身坐标、食物位置、分数、游戏状态



**关键挑战**：

- 用户态程序如何访问 GPU framebuffer？
- 如何实现低延迟的键盘输入响应？
- 如何在多任务环境中共享 GPU 资源？
- 如何避免用户态程序直接操作硬件带来的安全问题？

## **二、设计思路：用户态图形框架的三大支柱**



### **2.1 架构概览**



```
┌─────────────────────────────────────────────────────────────┐
│                    用户态贪吃蛇游戏                          │
│  ┌──────────────────────────────────────────────────────┐   │
│  │  游戏逻辑：蛇移动、碰撞检测、分数计算                 │   │

│  │  输入处理：读取键盘方向键                             │   │

└─────────────────────────────────────────────────────────────┘
```



### **2.2 核心设计原则**



**原则 1：最小化内核暴露面**

\- 内核只提供**必要**的硬件抽象（framebuffer 地址、大小、分辨率）

- 不暴露 GPU 寄存器细节，用户态通过 framebuffer 直接绘制
- 优点：简单、高效、易于理解

**原则 2：资源隔离与共享**

\- 多个任务可以同时运行，但**同一时刻只能有一个任务使用 GPU**

- 通过 `sys_yield()` 让出 CPU，避免忙等浪费
- 内核不管理 GPU 访问锁（简化设计，后续章节引入地址空间后再完善）

**原则 3：用户态自主渲染**

- 内核不实现任何渲染逻辑（如绘制矩形、文字）
- 用户态完全自主决定如何绘制（像素级控制）
- 优点：灵活性高，用户可以实现任意效果

## **三、系统调用设计：用户态与内核的契约**



### **3.1 系统调用接口设计**



需要新增 3 个自定义系统调用（ID 范围 2000-2099）：

字符码 (>0) 或 0（无输入）



### **3.2 设计思考题（启发式问题）**



**问题 1：为什么 framebuffer\_info 不直接返回指针，而是要求用户传入结构体地址？**



> 提示：考虑地址空间隔离。内核和用户态的虚拟地址空间不同，内核的物理地址对用户态没有意义。



**问题 2：为什么 key\_event 返回字符码而不是按键扫描码？**



> 提示：简化用户态处理逻辑。直接返回 ASCII 字符（'i'/'j'/'k'/'l'）比扫描码更直观。



**问题 3：gpu\_flush 为什么需要显式调用？内核能否自动刷新？**



> 提示：性能权衡。自动刷新会增加内核开销，用户态可以批量绘制后统一刷新。



**问题 4：如果两个任务同时调用 gpu\_flush，会发生什么？如何避免画面撕裂？**



> 提示：当前设计没有锁机制，可能导致画面混合。后续章节可以引入信号量或互斥锁。

### **4.2 关键实现步骤（提示性引导）**



#### **步骤 1：VirtIO GPU 探测与初始化**



**问题引导**：

- VirtIO 设备如何被内核发现？（MMIO 地址空间扫描）
- 如何判断设备类型是 GPU？（`transport.device_type() == DeviceType::GPU`）
- Framebuffer 的物理地址如何获取？（`gpu.setup_framebuffer()`）

**思考**：

- 为什么需要泄漏 GPU 驱动到 `'static` 生命周期？（全局访问需求）
- 为什么使用 `AtomicUsize` 存储 GPU 指针？（多任务并发访问）

#### **步骤 2：自定义系统调用分发**



**问题引导**：

- 如何区分标准系统调用（ID < 2000）和自定义系统调用（ID >= 2000）？
- `handle_syscall` 中如何处理自定义系统调用？（返回 `CustomSyscall` 事件）
- 自定义系统调用的参数如何传递？（从 `a0-a5` 寄存器读取）

**参考模式**：

```
match tg_syscall::handle(...) {
    Ret::Unsupported(_) => {
        let raw_id = self.ctx.a(7);
        if raw_id >= 2000 && raw_id <= 2099 {
            Event::CustomSyscall(raw_id, args)
        } else {
            Event::UnsupportedSyscall(id)
        }
    }
    // ...
}
```



#### **步骤 3：用户态上下文访问**



**问题引导**：

- 如何读取用户内存（`trace_request=0`）？（`core::ptr::read_volatile`）
- 如何写入用户内存（`trace_request=1`）？（`core::ptr::write_volatile`）
- 为什么需要 `unsafe`？（直接内存访问绕过 Rust 安全检查）

**安全提示**：

- 当前实现没有地址范围检查（后续章节引入地址空间后完善）
- 用户态传入非法地址会导致内核 panic（当前设计接受此风险）

## **五、用户态游戏实现：图形渲染与游戏逻辑**



### **5.1 Framebuffer 访问模式**



**用户态获取 framebuffer 信息**：

```
let mut fb_info = FramebufferInfo { fb_ptr: 0, fb_len: 0, width: 0, height: 0 };
sys_framebuffer_info(&mut fb_info);
let fb = unsafe { core::slice::from_raw_parts_mut(fb_info.fb_ptr as *mut u8, fb_info.fb_len) };
```



**问题引导**：

- 为什么需要 `unsafe`？（直接访问物理地址）
- 如何计算像素在 framebuffer 中的偏移？（`y * width + x`）
- 为什么每个像素占 4 字节？（RGBA 格式：R/G/B/Alpha）

### **5.2 渲染原语设计**



**问题引导**：

- 如何绘制一个矩形？（遍历像素，设置 RGBA 值）
- 如何绘制文字？（位图字体，每个字符 8x12 像素）
- 如何实现高效绘制？（避免重复绘制背景）

**设计建议**：

- 先实现基础原语：`draw_rect`、`draw_pixel`
- 再实现高级原语：`draw_snake`、`draw_food`、`draw_score`
- 最后实现游戏循环：输入处理 → 游戏逻辑 → 渲染 → flush

### **5.3 游戏循环与调度**



**问题引导**：

- 如何实现约 5 FPS 的更新频率？（读取 `rdtime`，比较时间差）
- 为什么需要 `sys_yield()`？（避免忙等浪费 CPU）
- `sys_yield` 与时间片轮转的关系？（协作式 + 抢占式结合）

**性能思考**：

- 如果不调用 `sys_yield()`，会发生什么？（其他任务无法执行）
- 如果时间片太短（如 1ms），对游戏有什么影响？（上下文切换开销大）
- 如果时间片太长（如 100ms），对游戏有什么影响？（游戏卡顿）

## **六、多任务并发：挑战与权衡**



### **6.1 并发场景分析**



**场景 1：贪吃蛇 + 文本任务并发**

```
App 0: Snake 游戏（图形渲染）
App 1: Power 计算（文本输出）
App 2: Write 测试（文本输出）
```



**问题引导**：

- 三个任务如何共享 CPU？（时间片轮转）
- GPU 如何在任务间切换？（当前任务绘制，切换后画面保持）
- 如果蛇正在绘制一半被中断，会出现什么现象？（画面撕裂）

**思考**：

- 如何减少画面撕裂？（双缓冲、绘制锁）
- 如何保证游戏流畅性？（提高游戏任务优先级？）

### **6.2 资源竞争问题**



**问题引导**：

- 两个贪吃蛇任务能否同时运行？（可以，但画面会混合）
- 如何实现 GPU 访问互斥？（需要内核支持锁机制）
- 为什么当前设计不实现锁？（简化教学，后续章节完善）

**扩展思考**：

- 如果实现 GPU 锁，应该放在内核还是用户态？（内核更安全）
- 如何避免死锁？（锁顺序、超时机制）

## **七、调试与测试策略**



### **7.1 调试技巧**



**技巧 1：分阶段验证**

1. 先验证 framebuffer\_info 能正确返回地址
2. 再验证直接写入 framebuffer 能看到颜色变化
3. 最后验证完整游戏逻辑

**技巧 2：日志定位问题**

- 内核日志：`log::info!("Framebuffer: {:#x}", ptr)`
- 用户态日志：通过 `sys_write` 输出到 UART

**技巧 3：QEMU GDB 调试**

- 内核：`cargo run --gdb`
- 用户态：在 `snake_main` 设置断点

### **7.2 常见问题排查**



**问题 1：framebuffer\_info 返回 -1**

- 检查 VirtIO GPU 是否被正确探测
- 检查 `setup_framebuffer` 是否成功

**问题 2：屏幕无显示**

- 检查 framebuffer 地址是否正确
- 检查 RGBA 值是否设置（特别是 Alpha 通道）

**问题 3：键盘无响应**

- 检查 UART 是否启用（`scounteren` 寄存器）
- 检查 `key_event` 返回值是否为 0

**问题 4：游戏卡顿**

- 检查是否忘记 `sys_yield()`
- 检查时间片间隔是否过短

## **十二、总结**



本扩展的核心思想是：



1\. **最小化内核**：内核只提供必要的硬件抽象，不干预用户态逻辑

2\. **用户态自主**：用户态完全控制渲染和游戏逻辑

3\. **简化设计**：接受一些不完美（如画面撕裂、无内存保护），聚焦核心概念
